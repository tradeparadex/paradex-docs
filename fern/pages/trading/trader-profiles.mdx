Paradex routes every order through one of two lanes — **Retail** or **Pro/API** — based on how the order is submitted, not a permanent label on the trader.

<CardGroup cols={2}>
  <Card title="Retail Lane" icon="fa-thin fa-user">
    Protected slow lane for UI and interactive API orders. Speed bumps, lower rate limits, Retail fee eligibility, queue priority, and [RPI](/trading/rpi) access.
  </Card>
  <Card title="Pro/API Lane" icon="fa-thin fa-rectangle-pro">
    Fast lane for standard API flow. No speed bumps, higher rate limits, but no Retail fee eligibility, queue priority, or RPI access.
  </Card>
</CardGroup>

## Why Order Lanes Exist

Retail users benefit from execution protections; API traders need low latency and high throughput. Separating these flows lets Paradex provide:

- Stronger protection and better pricing for Retail orders via [RPI](/trading/rpi)
- Lower latency and higher throughput for API traders
- Clearer matching rules and more predictable behavior for market makers

## Retail Lane

The protected slow lane for UI traders and interactive API orders.

- **300ms speed bump** on order submission and cancellation
- Subject to Retail rate limits
- Qualifies for [Retail trading fees](/trading/trading-fees)
- Eligible for [RPI](/trading/rpi) liquidity
- Queue priority at each price level

**Intended for:** Paradex web UI orders, interactive API orders (via interactive JWT), and non-latency-sensitive flow.

## Pro/API Lane

The fast lane for standard API flow.

- **No** speed bumps
- Higher rate limits (see table below)
- No Retail fee eligibility, queue priority, or RPI access

**Intended for:** standard API trading, market making, programmatic strategies, and latency-sensitive or high-throughput systems.

## How Orders Are Classified

All API orders default to the **Pro/API Lane**.

To route an API order through the Retail Lane, request a JWT with `token_usage=interactive`:

```bash
POST /v1/auth?token_usage=interactive
```

Orders submitted with this token receive full Retail Lane treatment — speed bumps, Retail rate limits, Retail fees, RPI access, and queue priority. Orders submitted without it are Pro/API Lane orders.

See the [Get JWT](/api/prod/auth/get-jwt) endpoint for details.

<Warning>
If an account exceeds Retail rate limits, it automatically switches to the Pro/API Lane and loses RPI access. This protects the Retail Lane from high-frequency order flow.
</Warning>

## Relationship to RPI

[Retail Price Improvement](/trading/rpi) is available only to Retail Lane orders. RPI orders are posted by liquidity providers to offer better pricing for Retail flow while staying protected from latency-sensitive API flow.

- RPI orders are visible in the UI but hidden from the API
- Only Retail Lane orders can match with RPI liquidity
- Liquidity providers can quote more aggressively because their RPI orders are shielded from standard API flow

## Relationship to FastFills

[FastFills](/trading/fastfills) is designed to incentivize Pro/API traders to take passive Retail Lane liquidity. When a Pro/API order matches a passive Retail Lane order, the Pro/API trader automatically receives a fee discount — improving fill speed for Retail users.

No special configuration is required. Qualifying fills carry the `fastfill` flag in fill responses.

## Implementation Guide

<Steps>
  <Step title="Choose the correct lane">
    **Retail Lane** — interactive, non-latency-sensitive orders that benefit from Retail protections.

    **Pro/API Lane** — latency-sensitive, programmatic, or high-throughput orders.
  </Step>
  <Step title="Request the correct JWT token">
    For Retail Lane API orders:

    ```bash
    POST /v1/auth?token_usage=interactive
    ```

    For Pro/API Lane orders, use a standard JWT token. See the [Get JWT](/api/prod/auth/get-jwt) endpoint for details.
  </Step>
  <Step title="Submit authenticated orders">
    Use the returned token for all authenticated requests. Interactive token → Retail Lane. Standard token → Pro/API Lane.
  </Step>
</Steps>
