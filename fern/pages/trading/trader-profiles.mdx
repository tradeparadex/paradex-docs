## Attributes

|            | <Badge intent="success" outlined minimal>Retail</Badge> | <Badge intent="note" outlined minimal>Pro</Badge> |
|------------|-|-|
| Speed Bump | Order Submission: 500ms <br /> Order Cancellation: 300ms | None |
| Rate Limit | 3 orders / sec <br /> 30 orders / min <br /> 300 orders / hr <br /> 1000 orders / 24 hrs | 800 orders / sec <br /> 17,250 orders / min |

<sub>Retail orders exceeding rate limits automatically switch to Pro trader profile and lose access to RPI liquidity.</sub>

For detailed fee information, see [Trading Fees](/trading/trading-fees).

## Identifying as Retail via API

By default, all orders submitted through the API are treated as <Badge intent="note" outlined minimal>Pro</Badge> orders. To have your API orders treated as <Badge intent="success" outlined minimal>Retail</Badge> orders, request a JWT token with the `token_usage=interactive` query parameter.

When using an interactive token:

- The order is subject to **Retail speed bumps** and **rate limits**
- The order qualifies for **reduced trading fees** (0% maker, 0.0075% taker)
- The order is eligible for **[Retail Price Improvement (RPI)](/trading/rpi)** liquidity
- The order receives **queue priority** at each price level

<Warning>
Orders placed with an interactive token are subject to Retail rate limits. If rate limits are exceeded, the account automatically switches to the Pro trader profile and loses access to RPI liquidity.
</Warning>

<Steps>
  <Step title="Get an interactive JWT token">
    Add `token_usage=interactive` when requesting a JWT:

    ```bash
    POST /v1/auth?token_usage=interactive
    ```

    See the [Get JWT](/api/prod/auth/get-jwt) endpoint for details.
  </Step>
  <Step title="Authenticate requests with the token">
    Use the returned token for all authenticated requests. Orders placed with this token are treated as <Badge intent="success" outlined minimal>Retail</Badge>.
  </Step>
</Steps>
