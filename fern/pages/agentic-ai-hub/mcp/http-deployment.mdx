# Deploying the HTTP Endpoint

This guide covers deploying the Paradex MCP server as a persistent HTTP endpoint.
A remote URL is required for **ChatGPT Desktop**, **Claude.ai web**, and any other client
that cannot run a local process.

## How it works

The server has two transport modes controlled by the `MCP_TRANSPORT` environment variable:

| Mode | Value | Use case |
|---|---|---|
| stdio (default) | `stdio` | Claude Desktop, Claude Code, Cursor — runs as a local subprocess |
| HTTP | `streamable-http` | ChatGPT, Claude.ai web, any remote client |

In HTTP mode the server listens on `MCP_PORT` (default `3000`) and exposes:
- `POST /mcp` — MCP protocol endpoint (what clients connect to)
- `GET /health` — health check, returns `{"status": "ok"}`

## Choosing a deployment option

| Option | Effort | Cost | Best for |
|---|---|---|---|
| [Railway](#railway) | Low | ~$5/mo | Quickest permanent setup |
| [Render](#render) | Low | Free tier available | Low-traffic / evaluation |
| [Fly.io](#flyio) | Medium | ~$2–5/mo | Global, low-latency |
| [AWS Lambda](#aws-lambda) | Medium | Pay-per-request | Scalable, zero idle cost |
| [Docker / VPS](#docker--vps) | High | Varies | Full control |

All options produce an endpoint of the form `https://your-domain.example.com/mcp`
that you paste into your AI client's MCP settings.

## Railway

The quickest path to a permanent URL. Railway auto-detects the `Dockerfile` and
handles TLS, routing, and restarts for you.

### Steps

1. Push your fork (or the upstream repo) to GitHub.

2. Go to [railway.app](https://railway.app) → **New Project → Deploy from GitHub repo**
   and select the repository.

3. Railway detects the `Dockerfile` automatically. Open **Variables** and add:

   | Key | Value |
   |---|---|
   | `MCP_TRANSPORT` | `streamable-http` |
   | `MCP_PORT` | `8080` |
   | `PARADEX_ENVIRONMENT` | `prod` |
   | `PARADEX_ACCOUNT_PRIVATE_KEY` | `your_private_key_here` |

4. Open **Settings → Networking → Generate Domain**.
   Your endpoint will be `https://<project>.up.railway.app/mcp`.

5. Paste that URL into ChatGPT Desktop: **Settings → Connectors → Create**.

### Notes

- Railway keeps the server running continuously — no cold starts.
- Redeploys happen automatically when you push to the connected branch.
- The free tier has usage limits; the Hobby plan ($5/mo) removes them.

## Render

Similar to Railway. Render has a free tier but free services spin down after 15 minutes
of inactivity and have a cold start on the next request (~30 s). Use a paid plan for
daily use.

### Steps

1. Go to [render.com](https://render.com) → **New → Web Service**.

2. Connect your GitHub repo. Render detects the `Dockerfile`.

3. Set the following under **Environment**:

   | Key | Value |
   |---|---|
   | `MCP_TRANSPORT` | `streamable-http` |
   | `MCP_PORT` | `8080` |
   | `PARADEX_ENVIRONMENT` | `prod` |
   | `PARADEX_ACCOUNT_PRIVATE_KEY` | `your_private_key_here` |

4. Set **Health Check Path** to `/health`.

5. Click **Create Web Service**. Your endpoint is
   `https://<service-name>.onrender.com/mcp`.

### Notes

- Free tier: service sleeps after 15 min of inactivity. First request after sleep
  will time out in most MCP clients. Upgrade to Starter ($7/mo) to avoid this.
- Render provides automatic TLS and a stable URL.

## Fly.io

More setup than Railway/Render but gives you fine-grained control over region,
scaling, and resources. Good choice if you want low-latency from a specific region.

### Steps

1. [Install `flyctl`](https://fly.io/docs/hands-on/install-flyctl/) and log in:

   ```bash
   brew install flyctl   # macOS
   fly auth login
   ```

2. From the repo root, run:

   ```bash
   fly launch --no-deploy
   ```

   Accept the generated `fly.toml`. When asked about regions, pick one close to you.

3. Set your secrets (stored encrypted, never in logs):

   ```bash
   fly secrets set \
     MCP_TRANSPORT=streamable-http \
     MCP_PORT=8080 \
     PARADEX_ENVIRONMENT=prod \
     PARADEX_ACCOUNT_PRIVATE_KEY=your_private_key_here
   ```

4. Open `fly.toml` and confirm the `[[services]]` block listens on port `8080`
   with the health check path set:

   ```toml
   [[services]]
     internal_port = 8080
     protocol = "tcp"

     [[services.tcp_checks]]
       grace_period = "10s"
       interval = "30s"
       restart_limit = 0
       timeout = "5s"

     [[services.http_checks]]
       interval = "10s"
       grace_period = "5s"
       method = "GET"
       path = "/health"
       protocol = "http"
       timeout = "2s"
   ```

5. Deploy:

   ```bash
   fly deploy
   ```

   Your endpoint will be `https://<app-name>.fly.dev/mcp`.

6. Paste the URL into ChatGPT Desktop: **Settings → Connectors → Create**.

### Notes

- Fly.io keeps one VM running by default (no cold starts on the free allowance).
- Use `fly scale count 0` to pause billing when not in use, `fly scale count 1` to resume.
- `fly logs` streams live logs for debugging.

## AWS Lambda

Serverless — no idle cost, scales automatically. The repo ships `Dockerfile.aws`
which bundles the [Lambda Web Adapter](https://github.com/awslabs/aws-lambda-web-adapter)
so no Lambda-specific code is needed. The MCP endpoint runs in stateless mode
(required because Lambda cannot hold open long-lived connections).

See the [AWS Lambda section in the README](https://github.com/tradeparadex/mcp-paradex-py#docker-aws-lambda--http) for
the full walkthrough.

**Key differences vs. the options above:**
- Set `MCP_STATELESS=true` (required for Lambda, skip this for all other options).
- Use `Dockerfile.aws`, not `Dockerfile`.
- Lambda Function URL invoke mode must be `RESPONSE_STREAM`.

## Docker / VPS

Run on any machine that can serve HTTPS — a DigitalOcean Droplet, Hetzner VPS, etc.

```bash
# Build
docker build . -t mcp-paradex-http

# Run
docker run -d \
  --name mcp-paradex \
  --restart unless-stopped \
  -p 8080:8080 \
  -e MCP_TRANSPORT=streamable-http \
  -e MCP_PORT=8080 \
  -e PARADEX_ENVIRONMENT=prod \
  -e PARADEX_ACCOUNT_PRIVATE_KEY=your_private_key_here \
  mcp-paradex-http
```

You'll need to put a reverse proxy (nginx, Caddy) in front to handle TLS.
ChatGPT requires `https://` — plain `http://` URLs are rejected.

**Caddy** is the simplest option for automatic TLS:

```
your-domain.com {
    reverse_proxy localhost:8080
}
```

```bash
caddy run --config /etc/caddy/Caddyfile
```

## Verifying the deployment

Once deployed, test the endpoint before connecting it to a client:

```bash
curl https://your-endpoint.example.com/health
# Expected: {"status":"ok"}

curl -X POST https://your-endpoint.example.com/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"tools/list","id":1}'
# Expected: JSON list of available tools
```

## Connecting to ChatGPT Desktop

1. Open ChatGPT Desktop → **Settings → Advanced → Developer Mode** (enable if not already).
2. Go to **Settings → Connectors → Create**.
3. Enter your endpoint URL (e.g. `https://your-domain.example.com/mcp`).
4. Click **I trust this provider**.

ChatGPT will immediately discover and list all available Paradex tools.

## Connecting to Claude.ai Web

1. Open [claude.ai](https://claude.ai) → **Settings → Integrations**.
2. Click **Add MCP Server**.
3. Enter the same endpoint URL.
